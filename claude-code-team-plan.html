Claude Code Team Plan

Productivity setup for a 12-developer team — React/nx frontends, Node serverless backends, individual Max plans.

12 devs · Max plan each 4–5 React apps (pnpm, nx) 3–5 Node serverless services July 2026
⚠️ The one billing fact to get right first Max plans are per-person and interactive — each dev's limits (5-hour rolling windows + weekly caps) are their own. That's perfect for daily dev work. But automated CI usage (auto PR-review bots) should not run on someone's personal Max token — it silently drains that one person's quota. If you ever add unattended automation, put it on a small dedicated Anthropic API key with a monthly spend cap. Everything interactive — including locally-run PR reviews — stays on the Max plans.

1 Foundation in every repo Week 1

Do this in each of the ~8–10 repos, and check it all into git so all 12 devs get it automatically:

CLAUDE.md

Run /init in each repo, then hand-curate to ~100 lines: build/test/lint commands, architecture in 5 bullets, conventions ("we use React Query, never raw fetch", "serverless handlers live in src/handlers, one file per function").

For the nx monorepo: a root CLAUDE.md plus small per-app CLAUDE.md files in apps/* — Claude picks up nested ones when working in those directories.

.claude/settings.json (checked in)

A shared permissions allowlist so nobody gets prompted for pnpm test, pnpm lint, nx affected, etc. After a week of real use, run /fewer-permission-prompts in each repo — it scans actual transcripts and generates the allowlist for you.

One CLAUDE.md owner per repo

When Claude repeatedly does something wrong, the fix goes in CLAUDE.md — not in everyone's individual prompts.

2 Skills, shared the right way Week 1–2

Yes, add skills — but don't copy-paste them into every repo. Use two layers:

Repo-specific skills — .claude/skills/ in that repo

  • new-component — scaffolds with your design system + tests + Storybook
  • new-endpoint — handler + validation + IaC entry
  • add-nx-lib, release

Good skills encode multi-step processes people get wrong, not things Claude already knows.

Org-wide skills — one internal plugin repo

Create a git repo that acts as a private plugin marketplace. Each dev adds it once (/plugin marketplace add your-org/claude-plugins) and gets shared skills, commands, and agents everywhere. Update centrally; everyone pulls the update.

This is the piece that scales to 12 people — otherwise skills drift across repos.

Start with 3–4 skills mapping to your most repetitive work. Resist writing 20 speculative ones — unused skills are just noise in context.

3 PR review Week 2

Local, before push

Make /code-review the norm before opening a PR. It uses the dev's own Max plan, catches bugs when they're cheapest to fix, and reduces reviewer load. Use /security-review for auth/payment/infra-touching changes.

On the PR — via local gh token (no GitHub Action needed)

claude-code-action is restricted in our org, so we skip it entirely. Claude Code uses the gh CLI: once authenticated (gh auth login or GH_TOKEN), /code-review <PR#> --comment reviews the PR and posts findings as inline comments directly on the pull request.

  • Workflow: the author runs it before requesting review, or the assigned reviewer runs it as their first pass.
  • Token: each dev uses their own gh auth (comments appear under their account). For a bot identity, create a machine user with a fine-grained PAT: Pull requests: Read & write + Contents: Read.
  • Billing: runs on the dev's own Max plan — no dedicated CI API key needed for this.
  • Later: standardize the review focus as a shared review-pr skill in the plugin repo (security for backend services, accessibility/perf for React apps).

4 Consistency via hooks Week 3

Add a checked-in PostToolUse hook that runs prettier --write (and optionally eslint --fix) on any file Claude edits.

This single hook eliminates the whole category of "Claude's formatting doesn't match ours" review comments across all 12 devs. Configure it in .claude/settings.json (the /update-config skill can set it up).

5 Team habits that stretch the Max plans

  • Plan mode for non-trivial work (shift+tab twice). Reviewing a plan costs almost nothing; letting Claude build the wrong thing burns a big chunk of a 5-hour window.
  • Model choice is the biggest lever on weekly limits. Sonnet for routine implementation; Opus/Fable for gnarly debugging and architecture. Switch per task with /model.
  • Scope prompts to the task — in the nx monorepo especially, telling Claude which app/lib to work in avoids expensive exploration.
  • A #claude-code Slack channel where devs post skills/prompts that worked. Recurring prompts graduate into skills in the shared plugin repo.
